O que o /audit faz

O /audit é uma skill de diagnóstico que varre a estrutura do seu AIOS e retorna um placar de 0 a 100. Ele lê — nunca escreve — todos os arquivos que compõem o sistema: manual, memória, skills, agents, MCPs, decisões e references. A pergunta central é simples: "o AIOS está bem construído?"

✓ O /audit faz

  • ✓Lê manual, memória, skills, agents e MCPs
  • ✓Pontua cada C de 0 a 25 (total 100)
  • ✓Lista lacunas priorizadas por alavancagem
  • ✓Salva audits/audit-{data}.md (opcional)

✗ O /audit NÃO faz

  • ✗Não escreve nenhum arquivo do sistema (read-only)
  • ✗Não executa skills ou agents durante a auditoria
  • ✗Não avalia a qualidade das automações em si
  • ✗Não sugere automações novas (isso é /level-up)

2

Estrutural, não capability

Existe uma distinção crítica que a maioria dos usuários confunde na primeira vez: o /audit mede se o AIOS está bem construído (estrutura), não se ele é capaz de fazer algo novo (capability). A pergunta "você poderia construir um brief diário automático?" é do /level-up, não do /audit.

🎯 A divisão fundamental

Estrutural (/audit)
  • → CLAUDE.md existe e tem seções preenchidas?
  • → Os 7 domínios de conexão estão cobertos?
  • → Existe pelo menos 1 trigger recorrente?
  • → Há pelo menos 3 skills funcionando?
Capability (/level-up)
  • → Deveria ter uma skill de brief semanal?
  • → Precisa de um agent de triagem de emails?
  • → Falta automação de atualização de CRM?
  • → Qual é a próxima automação de maior impacto?
💡
Dica prática

Se o /audit retornar lacunas como "você poderia construir X", anote e leve para o /level-up. O audit é um espelho de forma — não um consultor de estratégia. Usá-los na mesma sessão gera confusão de escopo.

/audit mede isso
Função
/level-up expande isso
Distinção
Não misture os dois

3

Como pontua os 4 Cs

Cada C é pontuado de 0 a 25. O /audit verifica critérios específicos em cada dimensão — não é subjetivo. Abaixo os critérios exatos que determinam seu score.

Critérios de pontuação — /audit
Context (0–25)
• CLAUDE.md existe e tem seções preenchidas +8
• Identidade/voz definida +4
• Memória (.claude/memory.md ou similar) +5
• References (.claude/references/) +4
• Decisões documentadas +4
Connections (0–25)
• Cobertura dos 7 domínios de conexão +14
• Guias de API / docs de ferramentas +4
• Freshness (última atualização < 30 dias) +4
• Balance read/write bem definido +3
Capabilities (0–25)
• 3+ skills ativas (.claude/skills/) +9
• 1+ skill própria (não do kit padrão) +8
• 1+ agent (.claude/agents/) +8
Cadence (0–25)
• Trigger recorrente configurado +10
• Atividade recente (últimos 7 dias) +8
• Templates de workflow +7
recriação ilustrativa dos critérios — valores exatos dependem da versão do kit

Stages — faixas de score

O score total (0–100) se enquadra em 4 stages. Cada stage tem um nome, uma faixa e uma leitura prática do estado do AIOS. A maioria dos projetos novos fica entre 40 e 70 — é o progresso esperado, não uma falha.

Score Stage Nome Leitura prática
0–39 Stage 0 Foundation AIOS incompleto. Peças essenciais faltando. Execute /onboard primeiro.
40–69 Stage 1 Built Estrutura funcional. Ainda há lacunas de impacto. Use /level-up toda semana.
70–89 Stage 2 Compounding AIOS sólido. Automações se retroalimentam. Foco em qualidade e cobertura.
90–100 Stage 3 Autonomous AIOS em plena operação. Score 95 é flex — sistema quase sem fricção manual.

Lacunas por alavancagem

O /audit não lista lacunas em ordem aleatória. Usa uma fórmula de alavancagem para priorizar o que resolver primeiro: leverage = pontos perdidos × multiplicador de impacto. As 3 lacunas de maior leverage chegam com next-step concreto.

Fórmula de alavancagem
leverage = pontos_perdidos × multiplicador
pontos_perdidos = 25 − score_do_C
multiplicador → depende do tipo de lacuna:
0 domínios Tier-1 (email, calendar, files) → ×4
CLAUDE.md raso / sem seções → ×3
sem skills próprias → ×2
tudo read-only (sem writes) → ×2
sem trigger recorrente → ×2
recriação ilustrativa da lógica de priorização

Exemplo: Top-3 lacunas priorizadas

1
Conexões Tier-1 ausentes leverage: 44

Email e calendar não conectados. Bloqueiam automações de maior ROI.

Next step: adicionar MCP Gmail + Google Calendar
2
CLAUDE.md raso leverage: 27

Seções de identidade e decisões quase vazias. Cada sessão começa do zero.

Next step: expandir seções Business, Voice, Decisions
3
Sem skills próprias leverage: 16

Só as 3 skills do kit. Nenhuma customizada para o negócio.

Next step: /level-up → criar 1 skill de brief semanal
💡
Não resolva tudo de uma vez

A lista de lacunas pode ter 8–12 itens. Foque apenas no Top-3 de leverage. Resolver o primeiro costuma elevar 10–15 pontos e mudar o ranking dos demais. Re-rode o /audit depois de cada melhoria.

Read-only + scoreboard

O /audit é estritamente read-only — nunca altera arquivos do sistema. O único efeito de escrita opcional é salvar o resultado em audits/audit-{data}.md para rastrear a evolução do score semana a semana. O formato usa barras ## a cada 5 pontos com labels descritivos.

7

Ritual semanal

O /audit é o gancho de compounding do AIOS. A primeira run cria o baseline no Dia 7. A partir daí, toda semana você roda, resolve o Top-1 de leverage e vê o placar subir. É o ciclo que transforma setup em sistema.

Ciclo semanal de auditoria

Dia 7
Primeira run — baseline

Execute /audit pela primeira vez. Aceite o score sem julgar. Salve em audits/audit-semana-1.md. Esse é o ponto zero.

Seg
Revisar o Top-1 da semana anterior

Abra o último audit.md. Qual foi o Top-1 de leverage? Foi resolvido? Anote o status antes de abrir nova sessão.

/audit
Rodar o /audit

Execute a skill. Novo score gerado. Compare com semana anterior. Quanto subiu? Novo Top-3 de leverage identificado.

→
Entregar para o /level-up

Pegue o Top-1 de leverage do /audit e leve direto para o /level-up. Uma lacuna → uma automação. Fechar o ciclo da semana.

💡
O efeito compounding é real

Resolver 1 lacuna de alta alavancagem por semana. Em 4 semanas, um AIOS Stage 1 (57 pts) costuma chegar a Stage 2 (75+ pts). O placar subindo é a evidência do sistema funcionando — não a automação individual.
